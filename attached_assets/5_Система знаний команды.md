Система знаний команды (поиск по своим данным + память)

Ценность

Даёт быстрые ответы по внутренним материалам и всегда показывает “на чём основан ответ” (источники) или честно говорит “данных недостаточно”.

Упрощение

Память/поиск можно сделать простым способом:- разбиение на абзацы + поиск по ключевым словам, или- векторное хранилище (если доступно и стабильно).

Техническое задание

Этот вариант — про “память команды”: когда информация уже где‑то есть, но её трудно быстро найти и проверить.

Кто пользователь (примеры):- новичок в команде: “где правила, ссылки, как у нас принято?”- менеджер/оператор: “как отвечать клиенту в такой ситуации?”- руководитель/тимлид: “как сделать так, чтобы знания не жили только в голове у одного человека?”Как выглядит проблема в реальной жизни:- ответы на одни и те же вопросы повторяются каждую неделю;- материалы раскиданы по файлам/документам, и поиск занимает время;- опасно “додумывать”: лучше либо показать источник, либо честно сказать “данных нет”.Что делает продукт:- хранит документы/заметки (ваша “база знаний”);- по вопросу ищет релевантные фрагменты;- формирует ответ только на основе найденных фрагментов и прикладывает цитаты‑источники;- если источников нет или уверенность низкая — включает ручную проверку (честный ответ “данных недостаточно”).Что считается ценностью:- экономия времени на поиск и онбординг;- единая точка правды (документы + история вопросов);- доказуемость: ответ сопровождается цитатами и аудитом.

3 сценария пользователя (обязательны)

Сценарий 1: пользователь загружает документы → они появляются в списке.

Сценарий 2: пользователь задаёт вопрос → получает ответ и цитаты‑источники.

Сценарий 3: пользователь задаёт вопрос, но подходящих фрагментов нет или уверенность низкая → получает ответ в духе “данных недостаточно”, метку needs_review=true и причину в истории/аудите.

Команды сервиса (точки доступа)

1) Добавить документ

- POST /kb/documents- назначение: добавить материал в базу знаний- вход: { "title": "string", "text": "string" }- выход: { "status": "ok", "document_id": "string" }- ошибки: 400/422 если title/text пустые или слишком длинные

2) Посмотреть документы (витрина)

- GET /kb/documents- назначение: показать список документов для веб‑панели- выход: список документов (id, title, created_at)

3) Задать вопрос системе

- POST /kb/ask- назначение: получить ответ по базе знаний (с источниками или с ручной проверкой)- вход: { "question": "string" }- выход: { "answer": "string", "sources": [{ "document_id": "string", "quote": "string" }], "needs_review": boolean }- требования:- если needs_review=false, то sources не пустой- если sources пустой, то обязательно needs_review=true

4) ИИ‑ответ с источниками (строгий JSON)

- POST /ai/answer_with_sources- назначение: отдельно показать качество ИИ‑операции (для тестов)- вход: { "question": "string", "context": "string" }- выход (строгий JSON):- answer: string- sources: { "quote": string }[]- confidence: "high"|"medium"|"low"- needs_review: boolean

Мини‑пример запросов для README (обязателен)

- curl для POST /kb/documents- curl для POST /kb/ask- curl для POST /ai/answer_with_sources

Когда нужна ручная проверка

Ручная проверка нужна, если:- источников нет (не нашли подходящий фрагмент);- confidence="low";- ответ требует домыслов.

Что должно произойти:- needs_review=true;- ответ в духе “данных недостаточно” (без выдумываний);- причина фиксируется в qa_runs.error и/или audit_runs.error.

База данных (SQLite)

Таблицы

Минимальные таблицы и назначение:- documents — документы базы знаний:- поля: id, created_at, title, text- snippets — фрагменты документов (поиск будет работать по ним):- поля: id, created_at, document_id, snippet_text- правило: можно резать по абзацам (простое разбиение достаточно)- qa_runs — история вопросов и ответов:- поля: id, created_at, question, answer, sources_json, needs_review, error- audit_runs — аудит всех команд сервиса:- audit_runs — аудит всех команд сервиса:- поля: id, created_at, action, input, output, status, error, duration_ms

Требование к данным

- sources_json хранит массив источников (document_id + quote);- если sources_json пустой, то needs_review=true.

Веб‑панель (3 раздела/вкладки)

Раздел 1 — “Документы”

Задача: управлять материалами базы знаний.

Что должно быть:- таблица документов (title, created_at, document_id)- форма добавления документа- действие: “Открыть документ” (показывает текст или первые N строк)

Раздел 2 — “Вопросы”

Задача: задать вопрос и получить ответ.

Что должно быть:- поле вопроса + кнопка “Спросить”- результат:- текст ответа- список источников (цитаты + ссылка на документ по document_id)- метка “требует проверки” если needs_review=true

Раздел 3 — “История”

Задача: видеть, что спрашивали и где система “не уверена”.

Что должно быть:- таблица последних 50–100 вопросов (created_at, question, needs_review)- фильтр: “требует проверки”- действие: открыть карточку вопроса (ответ + источники + причина ручной проверки)

Тестовые входы

Где хранить тесты

- tests_data/kb_documents.jsonl — документы

- tests_data/kb_questions.jsonl — вопросы

Формат строки документов:{"title":"...","text":"..."}Формат строки вопросов:{"question":"...","expected_needs_review":false,"why":"..."}

Набор документов

Сделайте 5 документов по 20–50 строк каждый. Примерные темы:- “Правила работы команды”- “Частые вопросы клиентов”- “Шаблоны ответов”- “Словарь терминов”- “Процесс запуска задачи”

Набор вопросов

- 7 вопросов, на которые ответ есть в документах → expected_needs_review=false и есть источники- 3 вопроса, которых нет в базе → expected_needs_review=true

Что написать в README рядом с тестами

Мини‑таблица:- № вопроса → ожидаемый expected_needs_review → почему → на какой документ опираемся (если опираемся).

Критерии приёмки

- документы добавляются и видны в витрине

- ответ содержит источники (цитаты) для 7 вопросов

- 3 вопроса приводят к needs_review=true и честному “данных недостаточно”

- аудит фиксирует запросы и причины ручной проверки
