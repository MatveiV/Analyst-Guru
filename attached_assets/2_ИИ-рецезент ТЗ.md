ИИ‑рецензент документов/технических заданий

Ценность

Помогает быстро понять качество документа: риски, пробелы, вопросы, критерии приёмки, чтобы подготовить документ к разработке.

Техническое задание

Для кого и какую проблему решаем

Пользователь: исполнитель, менеджер проекта, фрилансер или команда, которая получает “сырой текст” и должна понять:- что именно нужно сделать;- где риски и неопределённость;- что спросить у заказчика;- как выглядят критерии “готово”.Проблема: без рецензии ТЗ легко уехать в переработки и конфликт ожиданий.Решение: сервис превращает текст документа в структурированный отчёт и честно отмечает, когда без человека нельзя.

3 сценария пользователя (обязательны)

Сценарий 1: пользователь вставляет текст ТЗ → получает рецензию “в один клик”.

Сценарий 2: пользователь открывает рецензию → видит риски, вопросы и критерии приёмки (их можно копировать в рабочий документ).

Сценарий 3: пользователь видит метку “требует проверки” → понимает, что ИИ не уверен и нужны уточнения/ручная доработка.

Команды сервиса (точки доступа)

1) Создать документ

POST /documents

назначение: сохранить документ как “сырьё” для рецензии и истории

вход:

{"title":"string","text":"string"}

выход:

{"status":"ok","document_id":"string"}

ошибки:

400/422: пустой title, пустой text, слишком длинный text (задайте лимит, например 20–30 тыс. символов)

2) Запустить рецензию

POST /documents/{id}/review

назначение: создать рецензию (вызывая ИИ‑операцию), сохранить результат

выход: {"status":"ok","review_id":"string"}

требование: если документ не найден → 404

3) Витрина рецензий

GET /reviews/{id}

выход: { review_json, needs_review, created_at, document_id } (в удобном виде для веб‑панели)

4) ИИ‑рецензия (строгий JSON)

POST /ai/review

назначение: отдельно показать работу ИИ‑операции (для тестов и отладки качества)

вход: { "text": "string" }

выход (строгий JSON):

summary: string (2–6 предложений: о чём документ и что хотят сделать)

risks: `{ “severity”: “low”|“medium”|“high”, “description”: string }[] (минимум 3 риска, если есть данные)

missing_requirements: string[] (что не хватает, чтобы начать разработку)

questions_to_client: string[] (вопросы, которые снимут неопределённость)

acceptance_criteria: string[] (проверяемые критерии “готово”)

confidence: "high"|"medium"|"low"

needs_review: boolean

Когда нужна ручная проверка

Ручная проверка нужна, если:- confidence="low";- документ слишком сырой (1–3 строки, нет сути);- документ противоречивый.

Что должно быть в результате при ручной проверке:- needs_review=true;- questions_to_client не пустой (минимум 3 вопроса);- в audit_runs.error фиксируется причина (например: LOW_CONFIDENCE, TOO_VAGUE_INPUT, CONTRADICTORY_INPUT).

База данных (SQLite)

Таблицы

Минимальные таблицы:- documents: id, created_at, title, text- reviews: id, created_at, document_id, review_json, needs_review, error- audit_runs: id, created_at, action, input, output, status, error, duration_ms

Требование:- при создании рецензии сохраняйте полный JSON‑отчёт в review_json (как текст);- если ИИ вернул неверный формат → сохраняйте needs_review=true, error="INVALID_JSON" и отдавайте безопасный отчёт (хотя бы с вопросами).

Веб‑панель (3 раздела/вкладки)

Раздел 1 — “Документы”

список документов (минимум: title, created_at)

форма добавления документа (title + text)

действие: “создать рецензию” (запускает POST /documents/{id}/review)

Раздел 2 — “Рецензии”

список рецензий (по времени)

метка “требует проверки” (если needs_review=true)

действие: открыть рецензию

Раздел 3 — “Просмотр рецензии”

блоки отчёта:

краткое резюме (summary)

риски (с пометкой severity)

вопросы заказчику

критерии приёмки

экспорт: скачать JSON/CSV (минимум один формат)

Тестовые входы (10)

Формат тестов

Сделайте папку tests_data/specs/ или файл tests_data/specs.jsonl.Если используете .jsonl, то каждая строка:

{"title":"...","text":"..."}

Набор из 10 тестовых документов (примерный, можно адаптировать)

1–6) Нормальные мини‑ТЗ (по 5–20 строк):- “сделать форму заявки + таблицу результатов”- “парсер цен конкурентов”- “учёт оплат и выгрузка отчёта”- “личный кабинет с профилем”- “витрина данных + фильтры”- “мини‑сервис для генерации документов”

Ожидание: needs_review=false или true только если реально не хватает данных; есть 3+ риска и 5+ критериев приёмки.

7–8) Сырые/пустые- текст из 1–3 строк без конкретикиОжидание: needs_review=true и 3+ вопросов заказчику.

9–10) Противоречивые- в тексте одновременно “нужна авторизация” и “авторизация не нужна”, или “срок завтра” и “срок 2 месяца”Ожидание: needs_review=true или confidence="low", и среди рисков должен быть хотя бы один severity="high".

Что написать в README рядом с тестами (обязательно)

Мини‑таблица:- номер теста → ожидаемый needs_review → почему → что вы считаете “успехом”.

Критерии приёмки (для варианта B)

POST /documents сохраняет документ и возвращает document_id

POST /documents/{id}/review создаёт рецензию и возвращает review_id

рецензия содержит: summary, risks, questions_to_client, acceptance_criteria

минимум 2 теста из набора приводят к needs_review=true (сырые/противоречивые документы)

в веб‑панели видно метку “требует проверки” и можно открыть рецензию

в audit_runs видно 10+ запусков и причины ручной проверки/ошибок
