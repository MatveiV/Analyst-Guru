Выпускной проект по вайб‑кодингу

 Вы дошли до финального блока курса — выпускного проекта.

Если раньше вы делали отдельные блоки (API, база, интерфейс, ИИ, память, развёртывание), то сейчас задача другая: собрать цельный продукт, который:

решает понятную задачу;

работает стабильно на тестовых данных;

прозрачен (есть аудит и журнал ошибок);

воспроизводим (запуск по инструкции);

“не врёт уверенно”: умеет честно просить ручную проверку.

Это не “ещё одна практика”. Это ваш финальный кейс, который можно положить в портфолио и показывать как доказательство: “я умею доводить до результата”.

Что нужно сделать

Выберите один проект из пяти вариантов и реализуйте его как сквозную систему:

веб‑панель → серверная часть (API) → база данных → операция с языковой моделью (строгий JSON) → контроль качества → аудит → демонстрация.

Ограничение: не используем мессенджеры/соцсети как интерфейс (Telegram/WhatsApp/Instagram и аналоги). Интерфейс — только веб‑панель.

Почему этот выпускной проект важен?

Вы покажете не “ответ языковой модели”, а продукт, которому можно доверять:

Качество: строгий формат результата + проверка структуры + честная “ручная проверка”.

Управляемость: каждый запуск фиксируется в audit_runs, можно объяснить “почему так”.

Устойчивость: ошибки не ломают систему — они становятся видимыми и обработанными.

Воспроизводимость: проект запускается по README ≤ 10 минут и работает одинаково у вас и у другого человека.

Формула финала:

Проблема + Ограничения + Контекст + Результат + Защита + План внедрения + Мини‑экономика.

Общие требования (для любого варианта)

1) Серверная часть (API) — 3 точки доступа

В любом варианте должны быть 3 понятные точки доступа:

Точка 1 — “создать/запустить”: принимает входные данные и создаёт сущность в базе.

Точка 2 — “витрина”: отдаёт список сущностей (для таблицы в веб‑панели).

Точка 3 — “ИИ‑обработка”: вызывает языковую модель и возвращает строгий JSON по схеме.

2) База данных + аудит

Минимум SQLite:

- 2–4 доменные таблицы (под вариант);

- таблица audit_runs (или аналог), которая фиксирует каждый запуск:

- action, input, output, status, error, duration_ms, created_at.

3) “Ручная проверка” (обязательный элемент качества)

Это режим, когда система не делает вид, что уверена, а:

- выставляет needs_review=true;

- сохраняет причину (почему нужна проверка);

- показывает запись в веб‑панели в разделе “требует проверки”;

- возвращает безопасный результат без выдумываний.

4) Веб‑панель (3 экрана минимум)

Должно быть минимум:

- экран списка (витрина),

- экран карточки записи (детали + “сырой” ввод/вывод),

- экран аудита/ошибок/ручной проверки.

5) Мини‑экономика (обязательная часть отчёта)

Посчитайте “на салфетке”:

- время на 1 ручную операцию “до/после” (мин),

- стоимость 100 операций “до/после”,

- вывод: в чём ценность.

Варианты выпускного проекта (выберите 1):

Персональный помощник «от текста до действия». Сложность: **.

Система знаний команды (поиск по своим данным + память). Сложность: ***.
